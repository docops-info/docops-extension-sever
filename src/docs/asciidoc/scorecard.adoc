= ScoreCard Maker

image::icons/ScorecardIcon.svg[Scorecard Icon,80,80]

== Introduction

Scorecards create before/after comparisons that make improvements immediately scannable. Perfect for release notes, migration summaries, and transformation reports where you need to show what changed and why it matters.

=== Key Features

* *Before/After columns* - Side-by-side comparison for instant impact
* *Section grouping* - Organize items by category (features, bugs, security)
* *Visual indicators* - Red bullets (before) vs. green checkmarks (after)
* *Subtitle support* - Add context with subtitle line
* *Dark mode optimized* - Clean appearance in both light and dark themes

== Software Release Scorecard

Track features added and bugs resolved in a release:

[source,asciidoc]
....
[docops,scorecard]
----
title=Software Release v2.4.0 - Feature & Bug Summary
subtitle=Migration from Legacy System to Modern Architecture
---
[before]
title=BEFORE v2.4.0
---
[before.items]
=== Feature Status
Dark Mode Theme | Missing feature affecting user experience
Multi-language Support | Not available, limiting global reach
=== Known Issues
Login timeout issues | Users frequently logged out
Memory leaks in dashboard | System becomes slow over time
---
[after]
title=AFTER v2.4.0
---
[after.items]
=== New Features Added
Dark Mode Theme | Implemented with user preference saving
Multi-language Support | Added 12 languages with automatic detection
=== Bugs Resolved
Login timeout issues | Session management completely rewritten
Memory leaks in dashboard | React components optimized, memory usage -67%
----
....

== Common Use Cases

* *Release notes* - Highlight features shipped and bugs fixed in each release
* *Migration summaries* - Show before/after state when moving systems or platforms
* *Security audits* - Document vulnerabilities found and remediation completed
* *Process improvements* - Compare old workflow vs. new optimized process
* *Performance benchmarks* - Show metrics before and after optimization

== Format Options

=== Basic Structure

[source,asciidoc]
....
[docops,scorecard]
----
title=Main Title
subtitle=Optional Subtitle
---
[before]
title=BEFORE STATE
---
[before.items]
=== Section Name
Item Title | Item Description
Another Item | Another Description
---
[after]
title=AFTER STATE
---
[after.items]
=== Section Name
Improved Item | How it improved
New Item | What was added
----
....

=== Common Options

* `useDark=true` - Enable dark theme
* `scale=1.5` - Increase size (default: 1.0)
* `subtitle=Text` - Add explanatory subtitle below title

=== Section Headers

Use `===` to create section groupings:

[source]
----
=== Feature Status
Dark Mode | Implemented
Search | Enhanced

=== Known Issues
Bug 1 | Fixed
Bug 2 | Resolved
----

== Best Practices

* *Group related items* - Use section headers (===) to organize by category
* *Keep descriptions concise* - One line per item for scanability
* *Lead with impact* - Put highest-impact changes first in each section
* *Use parallel structure* - Keep before/after sections aligned by category
* *Limit to 8-12 items per column* - Too many items lose visual impact

[TIP]
====
For release notes, group by type (Features, Bugs, Security) rather than severity. This makes it easier for different audiences (users vs. ops teams) to scan for what matters to them.
====
